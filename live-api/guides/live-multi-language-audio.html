Bedarf

  • Ein für die Live-API eingerichtetes Konto mit aktivierter mehrsprachiger Audiofunktion (Einzelheiten erfahren Sie von Ihrem Account Manager).
  • Ein API-Schlüssel für die Live-API (wird Ihnen bei der Einrichtung des Kontos zur Verfügung gestellt)
  • Ein Encoder, der das RTP-Protokoll unterstützt. Die Unterstützung von mehrsprachigem Audio ist auf beschränkt rtp oder rtp-fec Protokoll.
  • Die PID jeder Audiospur muss beim Erstellen des Jobs bekannt sein.

Erstellen Sie Ihren Live-Job

Um Ihrem Job mehrere Audiospuren hinzuzufügen, muss beim Erstellen eines Live-Jobs eine Liste der Audiospuren im Anforderungshauptteil bereitgestellt werden:

{
  "live_stream": true,
  "region": "us-west-2",
  "protocol": "rtp",
  "cidr_whitelist": [ /* omitted… */ ],
  "outputs": [ /* omitted… */ ],
  "alternate_audio": {
    "tracks": [ { Track Details } ]
  }
}

Unterstützte Felder für die Track Objekt

Die folgende Tabelle enthält eine vollständige Beschreibung der unterstützten Felder in der track Objekt

Track Fields
Feld Beschreibung
pid Erforderlich. Die PID, die aus dem eingehenden TS für die Spur ausgewählt werden soll, die in der an Wowza gesendeten Stream-Datei angegeben ist.
language Erforderlich. Der Code für die zu verwendende Sprache. Gegenwärtig ist dies flexibel und kann in jedem der verschiedenen Sprachformate vorliegen, die von den Spezifikationen benötigt / unterstützt werden, insbesondere RFC5456 / BCP47, das auch ISO-639 gemäß den Anforderungen abdeckt. https://tools.ietf.org/html/rfc5646

HLS> LANGUAGE Feld

DASH> AdaptationSet:lang Feld
variant Optional. Zuordnungen zu den Werten des DASH-Rollenschemas: Haupt-, Alternativ-, Kommentar-, Zusatz-, Synchronisationswerte. Wird in der DASH-Rolle verwendet urn:mpeg:dash:role:2011. Der Standardwert ist main.

HLS> EIGENSCHAFTEN siehe https://tools.ietf.org/html/rfc8216#section-4.3.4.1
label Optional. Eine Textbeschreibung, die für den Track verwendet werden soll und vorzugsweise vom Spieler verwendet werden soll, sofern dies möglich ist. Benutzt in: HLS> NAME Feld. Der Standardwert ist language mit einer ID für jede Spur, z en-0 falls nicht angegeben.
streams[] Optional. Wenn keine Kopie vorhanden ist, verwenden Sie nur die Standard-Audioeinstellungen.

HLS> GROUP-ID. (Objekt)

Beispiel: { “source”: “720p” } - Beschriftung der Ausgabequelle abgleichen.

Dies ahmt das nach streams Feld des Wiedergabelistenausgabetyps.
default Erforderlich, außer wo Standard. Welche Audiospur soll als markiert werden? DEFAULT in dem EXT-X-MEDIA sowie welche in den Videostream gemuxt werden sollen *

Wenn nicht vorhanden, ist die erste Standardeinstellung.

* Wenn mehrere Wiedergabelisten mit unterschiedlichen Standardeinstellungen definiert sind, ist der Standardtitel der obersten Ebene der gemuxte

Konfigurieren Sie Ihren Encoder

Konfigurieren Sie Ihren Encoder so, dass MPEG-TS über RTP aus der Erstellungsantwort an stream_url gesendet wird. Stellen Sie sicher, dass die im Anforderungshauptteil bereitgestellten Audiospuren mit der richtigen PID in Ihrem Encoder konfiguriert sind.

Testen Sie die Wiedergabe

Laden Sie die playback_url in die Brightcove-Player oder Safari, um die Wiedergabe zu bestätigen.

DRM_Playback

Derzeit wird DRM nur über den Catch-up-Dienst (Timeshift) unterstützt. Fügen Sie diese Felder zu Ihrem Anforderungshauptteil "Jobanforderung erstellen" hinzu:

{
  "drm": {
    "modes": [
      "all"
    ]
  },
  "timeshift": {
    "type": "single",
    "startover_duration": 36000,
    "skip_live_output": true
  }
}

Die Antwort enthält DRM-Ausgänge und a timeshift_url zur Wiedergabe verwenden.
